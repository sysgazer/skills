# Best Practice Guide Generator Reference

## Overview

The Best Practice Guide Generator analyzes project documentation to identify knowledge gaps, then uses web search (when available) to find authoritative best practices and generate guides stored as either documentation links or standalone skills. This creates a self-improving documentation system where agents can discover needed context without loading large documents.

## Methodology

### Phase 1: Documentation Analysis

#### 1.1 Target Document Identification
- **Primary targets**: `AGENTS.md`, `CLAUDE.md`, `README.md`
- **Secondary targets**: `SPECS.md`, `CONTRIBUTING.md`, `DEVELOPMENT.md`
- **Configuration files**: `package.json`, `pyproject.toml`, `Cargo.toml`, etc.
- **Source code patterns**: Directory structure, naming conventions, technology choices

#### 1.2 Gap Detection Techniques
- **Presence analysis**: What topics are mentioned vs. what's explained in detail
- **Completeness scoring**: How thoroughly each topic is covered (shallow vs. deep)
- **Cross-reference analysis**: Are concepts properly linked and explained?
- **Recency checking**: When was documentation last updated vs. technology changes?
- **Consistency checking**: Do different documents provide consistent guidance?

#### 1.3 Impact Assessment Criteria
- **Security impact**: Gaps that could lead to security vulnerabilities
- **Performance impact**: Missing optimization guidance
- **Maintenance impact**: Undocumented patterns causing technical debt
- **Onboarding impact**: Missing context for new contributors/agents
- **Compliance impact**: Missing regulatory or standards guidance

### Phase 2: Research Protocol

#### 2.1 Search Strategy
- **Primary queries**: "[Technology] best practices [Year]"
- **Secondary queries**: "[Technology] common pitfalls", "[Technology] security considerations"
- **Source prioritization**:
  1. Official documentation (framework/library websites)
  2. Authoritative industry sources (Google, Microsoft, AWS, etc. blogs)
  3. Recognized expert blogs and whitepapers
  4. Community resources (Stack Overflow, GitHub discussions)
  5. Academic papers (for advanced topics)

#### 2.2 Source Validation
- **Authority assessment**: Author/organization credibility
- **Recency assessment**: Publication date vs. technology changes
- **Consistency checking**: Multiple sources agreeing on recommendations
- **Practical applicability**: Recommendations that work in real projects
- **Bias detection**: Commercial vs. objective recommendations

#### 2.3 Information Synthesis
- **Extract core principles**: Fundamental best practices
- **Identify patterns**: Common solutions to common problems
- **Document trade-offs**: When to use which approach
- **Provide examples**: Concrete implementation patterns
- **Note exceptions**: When rules don't apply

### Phase 3: Guide Generation

#### 3.1 Content Structure
```
1. Overview & Purpose
   - What problem does this guide solve?
   - Who is this guide for?
   - When to use these practices

2. Core Principles
   - Fundamental rules that should always be followed
   - Rationale for each principle
   - Consequences of violating principles

3. Implementation Patterns
   - Specific code/configuration examples
   - Step-by-step implementation guides
   - Common variations and when to use them

4. Pitfalls to Avoid
   - Common mistakes and how to recognize them
   - Anti-patterns and why they're problematic
   - Warning signs of improper implementation

5. Validation & Verification
   - How to confirm practices are being followed
   - Testing strategies for the practices
   - Metrics for success measurement

6. Additional Resources
   - Authoritative sources used
   - Related guides and documentation
   - Tools and utilities that help implementation
```

#### 3.2 Format Selection Criteria

**Choose Documentation Integration When:**
- Guides are highly project-specific
- Context is tightly coupled to project decisions
- Guides reference project-specific configurations
- Team prefers centralized documentation

**Choose Skill Creation When:**
- Guides are reusable across multiple projects
- Content represents general best practices
- Guides include executable scripts or tools
- Agents need to discover guides independently

**Choose Hybrid Approach When:**
- Core principles are general but implementation is project-specific
- Both agents and humans need different presentation formats
- Some content is stable (skills) while other content evolves (docs)

#### 4.3 Maintenance Procedures
- **Review schedule**: Quarterly review of generated guides
- **Update triggers**: Technology version changes, security alerts, pattern changes
- **Validation checks**: Automated tests for guide accuracy
- **Feedback collection**: User/agent feedback on guide usefulness
- **Archival process**: Retiring outdated guides with migration paths

## Implementation Examples

### Example 1: React Performance Guide Gap

**Detection:**
- `AGENTS.md` mentions React but no performance guidance
- `package.json` shows React 18+ but no React 18 best practices
- Code patterns show inefficient re-renders

**Research:**
- Search: "React 18 performance best practices 2026"
- Sources: React official docs, React performance workshops, industry blogs
- Findings: New React 18 features (concurrent features, transitions) require new patterns

**Guide Generation (as Skill):**
```
skills/react-performance-guide/
├── SKILL.md (with React 18 specific patterns)
├── references/README.md (detailed explanations)
└── scripts/analyze-renders.sh (performance analysis tool)
```

**Documentation Integration:**
```markdown
## React Performance Best Practices

For comprehensive React 18 performance guidance, see `skills/react-performance-guide/`.

Key updates for React 18:
- Use `startTransition` for non-urgent updates
- Implement proper Suspense boundaries
- Optimize re-renders with `React.memo` and `useMemo`

*Generated by best-practice-guide skill - Last updated: 2026-02-26*
```

### Example 2: Database Security Guide Gap

**Detection:**
- `CLAUDE.md` mentions database but no security guidance
- Project handles sensitive user data
- No documentation on encryption, access controls, or auditing

**Research:**
- Search: "database security best practices [database type]"
- Sources: OWASP database security guide, database vendor docs, compliance requirements
- Findings: Specific patterns for encryption at rest, connection security, audit logging

**Guide Generation (as Documentation):**
```markdown
## Database Security Guidelines

### Encryption
- Always encrypt sensitive data at rest
- Use TLS for all database connections
- Implement proper key rotation procedures

### Access Controls
- Principle of least privilege for database users
- Regular access review and cleanup
- Separate application and administrative access

### Monitoring & Auditing
- Enable database audit logging
- Monitor for suspicious access patterns
- Regular security scanning of database configurations

*Generated by best-practice-guide skill - Sources: OWASP, [Database Vendor] docs*
```

## Search Protocol

### When Search Tools Are Available:
1. **Always search** for authoritative sources before generating guides
2. **Cross-reference** at least 2-3 independent authoritative sources
3. **Document sources** with URLs and dates accessed
4. **Prioritize recency** for fast-changing technologies
5. **Note conflicting advice** and explain rationale for chosen approach

### When Search Tools Are Not Available:
1. **Acknowledge limitation** in generated guides
2. **Use existing knowledge** from similar projects
3. **Flag for later research** when tools become available
4. **Create placeholder structure** that can be filled later
5. **Focus on project-specific** patterns that don't require external research

### Search Query Patterns:
- `"[technology] best practices [current year]"`
- `"[technology] security considerations"`
- `"[technology] performance optimization"`
- `"[technology] common mistakes to avoid"`
- `"[technology] [specific use case] patterns"`

## Quality Metrics

### Guide Effectiveness:
- **Usage frequency**: How often guides are referenced
- **Clarity score**: User/agent feedback on understandability
- **Completeness**: Percentage of relevant topics covered
- **Accuracy**: Error reports or corrections needed
- **Timeliness**: Guide recency vs. technology changes

### Process Effectiveness:
- **Gap detection accuracy**: Percentage of real gaps identified
- **Research efficiency**: Time to find authoritative sources
- **Guide generation time**: Time from gap detection to usable guide
- **Maintenance burden**: Effort required to keep guides current
- **Agent adoption**: How well agents use generated guides

## Ethical Considerations

### Intellectual Property:
- Always attribute sources properly
- Respect copyright and licensing
- Use open source and freely available resources
- Create original synthesis rather than copying

### Accuracy & Responsibility:
- Clearly distinguish between facts and opinions
- Note when advice is controversial or debated
- Warn about potential risks of recommendations
- Encourage verification of critical guidance

### Bias Awareness:
- Seek diverse sources to avoid single-perspective bias
- Acknowledge trade-offs between different approaches
- Consider different project contexts and constraints
- Avoid promoting specific commercial products without alternatives

## Continuous Improvement

### Feedback Loop:
1. **Monitor guide usage** by agents and humans
2. **Collect feedback** on guide effectiveness
3. **Identify new gaps** through usage patterns
4. **Update guides** based on feedback and new information
5. **Measure improvement** in project outcomes

### Metrics Collection:
- Guide access frequency and duration
- User satisfaction surveys
- Error reports and correction requests
- Guide citation in other documentation
- Impact on project quality metrics

### Evolution Patterns:
- **Simple guides** → **Comprehensive skill suites**
- **Project-specific** → **Reusable patterns**
- **Static documentation** → **Interactive tools**
- **Human-focused** → **Agent-optimized formats**
